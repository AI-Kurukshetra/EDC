Deployment:

────────────────────────────────────────────────────────────────────────────────
SUPABASE DATABASE SCHEMA
────────────────────────────────────────────────────────────────────────────────

Create the following tables with full RLS policies. All tables must have:
  - id UUID PRIMARY KEY DEFAULT gen_random_uuid()
  - created_at TIMESTAMPTZ DEFAULT now()
  - updated_at TIMESTAMPTZ DEFAULT now()

--- USERS & AUTH ---

profiles (extends auth.users)
  - id UUID REFERENCES auth.users(id) ON DELETE CASCADE
  - full_name TEXT NOT NULL
  - email TEXT NOT NULL UNIQUE
  - role TEXT NOT NULL CHECK (role IN ('super_admin','sponsor','investigator',
    'coordinator','monitor','data_manager','read_only'))
  - is_active BOOLEAN DEFAULT true

--- STUDIES ---

studies
  - title TEXT NOT NULL
  - protocol_number TEXT UNIQUE NOT NULL
  - phase TEXT CHECK (phase IN ('Phase I','Phase II','Phase III','Phase IV'))
  - status TEXT DEFAULT 'draft' CHECK (status IN
    ('draft','active','on_hold','completed','terminated'))
  - sponsor_id UUID REFERENCES profiles(id)
  - description TEXT
  - target_enrollment INT
  - start_date DATE
  - end_date DATE
  - therapeutic_area TEXT

--- SITES ---

sites
  - study_id UUID REFERENCES studies(id) ON DELETE CASCADE
  - name TEXT NOT NULL
  - site_code TEXT NOT NULL
  - principal_investigator_id UUID REFERENCES profiles(id)
  - address TEXT
  - country TEXT
  - status TEXT DEFAULT 'pending' CHECK (status IN
    ('pending','active','closed'))

site_users (many-to-many: sites <> profiles)
  - site_id UUID REFERENCES sites(id) ON DELETE CASCADE
  - user_id UUID REFERENCES profiles(id) ON DELETE CASCADE
  - role TEXT NOT NULL
  - PRIMARY KEY (site_id, user_id)

--- SUBJECTS ---

subjects
  - study_id UUID REFERENCES studies(id) ON DELETE CASCADE
  - site_id UUID REFERENCES sites(id)
  - subject_id TEXT NOT NULL             -- e.g. "001-001"
  - status TEXT DEFAULT 'screened' CHECK (status IN
    ('screened','enrolled','randomized','completed','withdrawn','screen_failed'))
  - consent_date DATE
  - enrollment_date DATE
  - withdrawal_date DATE
  - withdrawal_reason TEXT
  - UNIQUE (study_id, subject_id)

--- FORMS (CRF BUILDER) ---

form_templates
  - study_id UUID REFERENCES studies(id) ON DELETE CASCADE
  - name TEXT NOT NULL                   -- e.g. "Demographics", "Adverse Events"
  - form_type TEXT CHECK (form_type IN
    ('screening','enrollment','visit','adverse_event','completion','other'))
  - version INT DEFAULT 1
  - is_published BOOLEAN DEFAULT false
  - schema JSONB NOT NULL               -- field definitions (see below)
  - visit_schedule JSONB               -- which visits this form applies to

-- schema JSONB structure:
-- {
--   "fields": [
--     {
--       "id": "field_uuid",
--       "label": "Date of Birth",
--       "type": "date|text|number|select|multiselect|radio|checkbox|textarea",
--       "required": true,
--       "validation": { "min": null, "max": null, "pattern": null },
--       "options": ["Yes","No"],     -- for select/radio types
--       "condition": {               -- show/hide logic
--         "field_id": "...",
--         "operator": "equals|not_equals|greater_than|less_than",
--         "value": "..."
--       }
--     }
--   ]
-- }

--- DATA ENTRIES ---

data_entries
  - subject_id UUID REFERENCES subjects(id) ON DELETE CASCADE
  - form_template_id UUID REFERENCES form_templates(id)
  - visit_number INT
  - visit_date DATE
  - data JSONB NOT NULL              -- { "field_id": value, ... }
  - status TEXT DEFAULT 'draft' CHECK (status IN
    ('draft','submitted','locked','sdv_required','sdv_complete'))
  - submitted_by UUID REFERENCES profiles(id)
  - submitted_at TIMESTAMPTZ
  - locked_by UUID REFERENCES profiles(id)
  - locked_at TIMESTAMPTZ

--- QUERIES ---

queries
  - data_entry_id UUID REFERENCES data_entries(id) ON DELETE CASCADE
  - field_id TEXT NOT NULL           -- references field within the form
  - subject_id UUID REFERENCES subjects(id)
  - query_text TEXT NOT NULL
  - status TEXT DEFAULT 'open' CHECK (status IN
    ('open','answered','closed','cancelled'))
  - raised_by UUID REFERENCES profiles(id)
  - assigned_to UUID REFERENCES profiles(id)
  - priority TEXT DEFAULT 'normal' CHECK (priority IN ('low','normal','high'))

query_responses
  - query_id UUID REFERENCES queries(id) ON DELETE CASCADE
  - response_text TEXT NOT NULL
  - responded_by UUID REFERENCES profiles(id)
  - action_taken TEXT

--- AUDIT LOG ---

audit_logs
  - study_id UUID REFERENCES studies(id)
  - user_id UUID REFERENCES profiles(id)
  - action TEXT NOT NULL            -- e.g. 'DATA_ENTRY_SUBMITTED'
  - entity_type TEXT NOT NULL       -- e.g. 'data_entry', 'subject', 'query'
  - entity_id UUID
  - old_value JSONB
  - new_value JSONB
  - ip_address TEXT
  - user_agent TEXT
  -- NOTE: NO UPDATE or DELETE permissions on this table ever. Insert only.

--- ELECTRONIC SIGNATURES ---

signatures
  - study_id UUID REFERENCES studies(id)
  - entity_type TEXT NOT NULL
  - entity_id UUID NOT NULL
  - signed_by UUID REFERENCES profiles(id)
  - signature_meaning TEXT NOT NULL  -- e.g. "I certify this data is accurate"
  - signed_at TIMESTAMPTZ DEFAULT now()
  - certificate_hash TEXT NOT NULL   -- SHA-256 of (entity_id + user_id + timestamp)

--- DOCUMENTS ---

study_documents
  - study_id UUID REFERENCES studies(id) ON DELETE CASCADE
  - name TEXT NOT NULL
  - file_path TEXT NOT NULL          -- Supabase Storage path
  - file_size INT
  - mime_type TEXT
  - version INT DEFAULT 1
  - uploaded_by UUID REFERENCES profiles(id)
  - category TEXT CHECK (category IN
    ('protocol','icf','investigator_brochure','sap','other'))

────────────────────────────────────────────────────────────────────────────────
ROW LEVEL SECURITY (RLS) RULES
────────────────────────────────────────────────────────────────────────────────

Apply these RLS rules to every table:

1. profiles        — Users can read all profiles; update only their own
2. studies         — Sponsor sees own studies; investigator/coordinator/monitor
                     see studies they are assigned to via site_users
3. sites           — Visible to users assigned to that site or study sponsor
4. subjects        — Visible to users at the subject's site + sponsor + DM
5. form_templates  — Readable by all study members; writable by DM/sponsor only
6. data_entries    — Coordinators at the site can insert/update their subjects;
                     monitors and DM can read all; no one can delete
7. queries         — Visible to all study members; coordinators can respond;
                     DMs can open/close
8. audit_logs      — INSERT only via Edge Function service_role; read by
                     DM, sponsor, monitor; NO UPDATE or DELETE ever
9. signatures      — INSERT only; read by all study members; never editable
10. study_documents— Readable by all study members; uploadable by DM/sponsor

EDGE FUNCTIONS TO BUILD
────────────────────────────────────────────────────────────────────────────────

supabase/functions/audit-log/index.ts
  - Accepts: { user_id, action, entity_type, entity_id, old_value, new_value }
  - Writes to audit_logs using service_role key (bypasses RLS)
  - Called from Next.js server actions for every state-changing operation

supabase/functions/generate-queries/index.ts
  - Triggers after a data_entry is submitted
  - Reads the form_template schema and validates each field value
  - Automatically inserts a query for any value outside defined validation range
  - Sends email notification to assigned coordinator via Resend

supabase/functions/export-data/index.ts
  - Accepts: { study_id, format, site_ids[], form_ids[], date_range }
  - Queries and joins data across subjects, data_entries, form_templates
  - Generates CSV or JSON output, uploads to Supabase Storage
  - Returns a signed download URL valid for 24 hours
  - Logs the export to audit_logs

supabase/functions/send-notification/index.ts
  - Accepts: { user_id, type, title, message, entity_id }
  - Inserts to notifications table
  - Sends email if the notification type warrants it (via Resend API)

────────────────────────────────────────────────────────────────────────────────
KEY IMPLEMENTATION PATTERNS
────────────────────────────────────────────────────────────────────────────────

1. SERVER COMPONENTS vs CLIENT COMPONENTS
   - Use React Server Components (RSC) for all initial data fetching
   - Use 'use client' only for interactive components (forms, charts, real-time)
   - Fetch initial data in page.tsx using the server Supabase client
   - Pass data as props to client components; they subscribe to Supabase
     Realtime for live updates

2. SERVER ACTIONS
   - All data mutations (insert, update) must go through Next.js Server Actions
   - Server actions call the audit-log Edge Function before returning
   - Validate with Zod on both client (for UX) and server (for security)

3. SUPABASE TYPE SAFETY
   - Generate types: npx supabase gen types typescript --local > types/database.types.ts
   - Use generated Database type everywhere; no 'any' types
   - Create typed helpers in lib/supabase/ for common query patterns

4. ERROR HANDLING
   - All Server Actions return: { data: T | null, error: string | null }
   - Client components handle error state with toast notifications
   - Add error boundaries at the route segment level

5. DYNAMIC CRF RENDERING
   - FormRenderer component accepts a form_template.schema JSONB
   - Renders each field using a field type switch statement
   - Conditional logic evaluated via a pure function on every field value change
   - Validation schema built dynamically from field definitions using Zod

- NEVER use 'any' TypeScript type. Use 'unknown' and narrow it, or generate
